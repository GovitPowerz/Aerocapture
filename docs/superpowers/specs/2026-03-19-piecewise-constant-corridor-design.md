# Piecewise-Constant Bank Guidance & Corridor Optimization

**Date:** 2026-03-19
**Status:** Approved

## Problem

The current corridor visualization and reference trajectory have two weaknesses:

1. **Legacy reference trajectory** — Energy Controller, PredGuid, FNPAG, and FTC all track a pre-recorded constant-bank trajectory (`msr_aller.dat`) from the original Fortran simulator. This reference was hand-picked, not optimized, and is static across training runs.

2. **Narrow restricted corridor** — The ±δZa restricted corridor is computed from constant-bank trajectories that span only ~2-3° of bank angle, producing a corridor ~0.04 kPa wide in the energy-pdyn plane — nearly invisible on plots.

## Solution

Add a new `piecewise_constant` guidance scheme: 10 constant bank angle segments uniformly distributed over orbital energy, GA-optimized for minimum total DV (including inclination correction). This serves two purposes:

1. **Produces an optimized reference trajectory** for other guidance schemes to track.
2. **Builds rich corridor envelopes** from the GA population history across all generations.

## Design

### 1. New Guidance Scheme — `piecewise_constant`

**Concept:** The orbital energy range is divided into 10 uniform segments. Each segment has a constant bank angle in [-180°, +180°]. Negative bank angles implicitly encode roll reversals without any lateral guidance logic. The trajectory is fully determined by the 10 bank values — pure open-loop, no navigation, no guidance feedback.

**Rust side** (`config.rs`, `runner.rs`):

In the reference trajectory code path in `runner.rs`, if `bank_angles` array is present in config, the sim:
1. Computes current orbital energy from the state vector.
2. Determines which of the 10 energy segments it falls in.
3. Commands that segment's bank angle (sign included).

TOML config:
```toml
[guidance]
type = "piecewise_constant"
reference_trajectory = true
bank_angles = [60.0, 65.0, 70.0, -65.0, -60.0, 55.0, 50.0, 45.0, -40.0, 35.0]
energy_range = [-6.0, 5.0]  # MJ/kg, auto-computed from mission data if absent
```

No roll reversal logic, no navigation, no pilot dynamics on bank sign — the bank angle (with sign) is directly commanded from the piecewise-constant profile.

**Python side** (`param_spaces.py`):

```python
"piecewise_constant": [
    ParamSpec("bank_angle_0", -180.0, 180.0, 65.0),
    ParamSpec("bank_angle_1", -180.0, 180.0, 65.0),
    ...
    ParamSpec("bank_angle_9", -180.0, 180.0, 65.0),
]
```

10 parameters, each in [-180°, +180°]. Same cost function as other schemes: total DV (including inclination) + TOML-configurable g-load/heat flux soft penalties.

**Evaluate** (`evaluate.py`): Decode chromosome → override `guidance.bank_angles` list in TOML overrides. The energy range is computed once from mission entry conditions and stored in the config.

### 2. Training Output Artifacts

When `--guidance piecewise_constant` training completes:

**Standard per-scheme outputs** (same as other schemes):
- `training_output/piecewise_constant/best_params.json` — the 10 bank angles
- `training_output/piecewise_constant/checkpoint.pkl`
- `training_output/piecewise_constant/training_log.jsonl`
- `training_output/piecewise_constant/convergence_report.html`
- `training_output/piecewise_constant/final_report.html`
- `training_output/piecewise_constant/final_report_corridors.png`

**Mission-level outputs** (shared across schemes):
- `training_output/<mission>/ref_trajectory.dat` — best individual's trajectory in the legacy 7-column format (energy, pdyn, radial_vel, altitude_rate, inclination, time, cos_bank). Generated by re-running the best individual with `include_trajectories=True` and extracting photo row columns.
- `training_output/<mission>/corridor_boundaries.npz` — 4 envelopes (crash, restricted upper, restricted lower, capture) built incrementally from all GA generations, plus Phase 2 deterministic boundary trajectories and nominal.

### 3. Reference Trajectory Dependency

Schemes that track the reference trajectory require it to exist before training:

```python
REQUIRES_REF_TRAJECTORY = {"energy_controller", "pred_guid", "fnpag", "ftc"}
```

At `train.py` startup, if `guidance_type in REQUIRES_REF_TRAJECTORY`:
- Check for `training_output/<mission>/ref_trajectory.dat`
- If missing, exit with explicit error:
  ```
  ERROR: No reference trajectory found for mission 'mars'.
  Run piecewise_constant training first:
    uv run python -m aerocapture.training.train --guidance piecewise_constant --toml <config>
  ```

The ref trajectory path is passed to the Rust sim via TOML override: `data.reference_trajectory = "training_output/<mission>/ref_trajectory.dat"`.

Schemes that don't need a reference (`equilibrium_glide`, `neural_network`, `piecewise_constant`) proceed without this check.

Mission name is derived from `[mission] planet` in the TOML config.

### 4. Incremental Corridor Envelope from GA History

During piecewise_constant training, each generation's evaluated trajectories feed an incremental envelope accumulator.

**`CorridorAccumulator` class** (in `corridor.py`):

```python
class CorridorAccumulator:
    energy_bins: np.ndarray          # (B,) fixed bin centers, set at init
    crash_max_pdyn: np.ndarray       # (B,) running max of non-crash trajectories
    restricted_max_pdyn: np.ndarray  # (B,) running max of corridor-classified
    restricted_min_pdyn: np.ndarray  # (B,) running min of corridor-classified
    capture_min_pdyn: np.ndarray     # (B,) running min of all captured
```

**Update protocol** — called once per generation:
1. `evaluate.py` calls `run_batch` with `include_trajectories=True`.
2. `classify_trajectories` classifies each trajectory (crash/undershoot/corridor/overshoot/hyperbolic) using `delta_za_restricted` from TOML.
3. For each trajectory, bin its (energy, pdyn) points and update the running max/min arrays.
4. Raw trajectory data is discarded — only the 4 envelope arrays persist.

**Energy range bootstrap:** Fixed at initialization from a single reference sim at nominal entry conditions. 200 bins.

**Smoothing:** Applied once at save time (not during accumulation) using `uniform_filter1d`.

**Checkpoint integration:** The 4 envelope arrays are saved/restored with the GA checkpoint, so training resume continues accumulating without losing prior generations.

### 5. Corridor Computation (Phase 2)

After piecewise_constant training completes, a deterministic Phase 2 runs (same as current):
- 10k bank-angle sweep [0°, 180°], no dispersions
- Extracts ±δZa boundary trajectories (dashed lines)
- Finds min-DV nominal (constant-bank reference)

This provides the dashed boundary lines and the constant-bank nominal for the visualization. The GA-accumulated envelopes provide the 4-layer fills.

Both are saved to `training_output/<mission>/corridor_boundaries.npz` (schema v3, extended with restricted envelopes).

### 6. Visualization — 4-Layer Fill

The corridor panel (a) in `final_report.py` uses 4-layer fills:

**Fill layers** (back to front):
1. Grey (`#BDBDBD`, alpha=0.5) above `restricted_max_pdyn` — undershoot zone
2. Red (`#E57373`, alpha=0.5) above `crash_max_pdyn` — crash zone (overpaints grey)
3. Grey (`#BDBDBD`, alpha=0.5) below `restricted_min_pdyn` — overshoot zone
4. Red (`#E57373`, alpha=0.5) below `capture_min_pdyn` — hyperbolic zone (overpaints grey)

**On top:**
5. Dashed grey lines — ±δZa boundary trajectories (Phase 2)
6. Blue spaghetti — MC guided trajectories (final evaluation)
7. Red line — nominal piecewise-constant (best GA individual)
8. Green line — nominal guidance scheme (min-DV from final eval)

White corridor between layers 1 and 3 = the restricted corridor where "good" captures happen.

**TOML config** (`[corridor]` section in mission TOML):
```toml
[corridor]
delta_za = 500.0            # km, for dashed boundary lines (Phase 2)
delta_za_restricted = 200.0  # km, for restricted corridor envelopes (GA)
n_sims = 10000              # Phase 2 deterministic sweep size
```

**Fallback behavior:**
- Corridor `.npz` exists → full 4-layer visualization
- Corridor `.npz` missing → spaghetti only, no zones
